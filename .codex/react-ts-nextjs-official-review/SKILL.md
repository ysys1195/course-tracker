React + TypeScript +
Next.js のコードベースで作業するときにこの skill を使ってください。  
目的は、リポジトリ内にある品質の低い既存パターンをそのまま踏襲するのではなく、React /
TypeScript /
Next.js の公式ガイドラインを優先して、コード生成・レビュー・リファクタを行うことです。

## 目的

- ローカルの弱い実装慣習よりも、公式ドキュメントの指針を優先する
- 保守しやすく、型安全で、レビューしやすいコードを出す
- React / TypeScript / Next.js の公式思想に沿った変更を行う

## React のルール

- React の Rules を厳守する
- Hooks はトップレベルでのみ呼び出し、React の関数内でのみ使う
- コンポーネントとカスタムフックは pure / idempotent に保つ
- 不要な Effect を避ける
- 描画のために導出できる値、単純なデータ変換、イベントハンドラ内に置ける処理のために`useEffect`を使わない
- state は矛盾・重複・冗長を避けた構造にする
- データフローが分かりやすくなるように、責務に応じてコンポーネントを分割する

## TypeScript のルール

- リポジトリ側で明確に無効化されていない限り、strict TypeScript を前提に考える
- `any` は原則避ける
- やむを得ず
  `any`を使う場合は、移行中・外部ライブラリ境界・型定義不足など理由が明確なときだけにする
- プリミティブ型は `String` / `Number` / `Boolean` ではなく、`string` / `number`
  / `boolean` を使う
- Utility Types を、意図の明確化や重複削減に役立つ場面で使う
- 公開される関数や重要な関数では、必要に応じて引数・戻り値の型を明示する
- 文脈からより狭いドメイン型を作れるなら、広すぎる曖昧な型を安易に使わない

## Next.js のルール

- App Router では、まず Server Components をデフォルトとして考える
- state、イベントハンドラ、ライフサイクル、ブラウザ API、クライアント専用フックが必要な場合だけ
  `"use client"` を付ける
- 秘密情報、サーバー側データアクセス、重い依存関係は可能な限りサーバー側に置く
- 不必要な client boundary を増やさず、クライアント JavaScript を最小化する
- ルーティング、レンダリング、パフォーマンス、型安全性については Next.js の production
  guidance に沿う

## レビュー時の振る舞い

- 既存コードをレビューする際は、ローカル慣習が公式ガイドラインと衝突している箇所を特定する
- 問題点は短く説明し、まずは最小で安全な改善案を出す
- 明示的に広いリファクタを求められていない限り、既存アーキテクチャはできるだけ壊さない

## 出力方針

- 変更はできるだけ小さく、焦点を絞った diff にする
- 設計上の判断が自明でない場合だけ、短く理由を書く
- 必要に応じて、そのコンポーネントが server のままでよいか、client にすべきかを明示する
- 適切であれば、lint / typecheck / test / build の確認手順も添える

## やってはいけないこと

- 既存コードにあるからという理由だけでアンチパターンを踏襲しない
- とりあえず `useEffect` を足さない
- 必要性がないのにファイル先頭へ `"use client"` を付けない
- 型エラーをごまかすためだけに `any` を乱用しない
